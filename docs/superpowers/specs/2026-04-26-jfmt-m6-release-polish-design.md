# jfmt M6 — Release Polish + cargo-dist (Design)

**Status:** approved 2026-04-26 (brainstormed with user)
**Predecessor specs:** Phase 1 design §8, §11; M5 design `docs/superpowers/specs/2026-04-25-jfmt-m5-schema-design.md`
**Predecessor plan:** `docs/superpowers/plans/2026-04-25-jfmt-m5-schema.md` (shipped as `v0.0.6`)
**Target tag:** `v0.1.0` (Phase 1 close, preview → 1.0-track)

## 1. Scope

M6 closes Phase 1 by turning the existing 0.0.x preview into a publishable
product:

1. **LICENSE files** — `LICENSE-MIT` + `LICENSE-APACHE` (workspace declares
   `MIT OR Apache-2.0` but no files exist).
2. **CHANGELOG.md** — keep-a-changelog format. `[0.1.0]` section is the
   detailed Phase 1 feature list. A short "Pre-1.0 history" subsection
   compresses v0.0.1–v0.0.5 to one line each.
3. **Criterion benchmarks** — three layered groups:
   - `bench_parser` — `EventReader::next_event` throughput on 1MB JSON.
   - `bench_writer` — `MinifyWriter` + `PrettyWriter` re-serialise 1MB.
   - `bench_ndjson_pipeline` — `run_ndjson_pipeline` on 100k-line NDJSON,
     `threads=1` vs `threads=cores`.
   Local `cargo bench` only; **not** integrated into CI.
4. **cargo-dist** — tag-triggered GitHub Releases workflow. Targets:
   `x86_64-unknown-linux-gnu`, `aarch64-unknown-linux-gnu`,
   `x86_64-apple-darwin`, `aarch64-apple-darwin`,
   `x86_64-pc-windows-msvc`. Tarballs + `shell` installer (curl|sh) +
   `powershell` installer. **No** crates.io publishing.
5. **Cargo.toml metadata** — every crate gets `description`, `keywords`,
   `categories`, `readme`. Workspace gets `homepage`, `documentation`.
6. **README polish** — CI badge, license badge, install via prebuilt
   binaries section.
7. **Tag `v0.1.0`** — Phase 1 closer. Manual verification of the first
   release.yml run (5 platform tarballs + 2 installer scripts on the
   GitHub Release page).

### 1.1 Out of scope (permanent Phase 1)

- crates.io publishing (separate Phase 1.5 decision).
- Bench CI integration / performance threshold gates.
- Windows MSI / macOS DMG installers (cargo-dist defaults to tarballs).
- Homebrew / AUR / winget / nixpkgs third-party indexes.
- `cargo audit` / `cargo deny` workflows.
- Dedicated documentation site (docs.rs covers it).

## 2. Design decisions

| # | Decision |
|---|---|
| D1 | Release channel = GitHub Releases via cargo-dist. No crates.io in M6. Reason: CLI users expect `curl | sh`; crates.io publishes are irreversible and we want the design to settle first. |
| D2 | Three layered benches (`parser`, `writer`, `ndjson_pipeline`) — not per-subcommand or single-canonical. Layering matches `jfmt-core`'s module boundaries; CLI overhead would muddy the regression signal. |
| D3 | CHANGELOG carries a detailed `[0.1.0]` Phase 1 feature list **plus** a one-line-per-tag "Pre-1.0 history" table — gives users the current product surface plus enough breadcrumbs to understand evolution. |
| D4 | Final tag is `v0.1.0`, not `v0.0.7`. Semver: pre-1.0 minor bump = "first non-preview release", a stronger stability promise than 0.0.x. |
| D5 | cargo-dist `pr-run-mode = "plan"` so PRs only validate; tags drive actual builds. Avoids accidental cross-platform builds on every PR. |

## 4. Bench design (criterion)

Each bench file follows this pattern:

```rust
use criterion::{black_box, criterion_group, criterion_main, Criterion};

fn fixture_1mb() -> Vec<u8> {
    // Build a deterministic ~1MB JSON array. In-process generation
    // avoids committing megabytes of fixtures to the repo.
    let mut buf = b"[".to_vec();
    let record = br#"{"id":12345,"name":"alice","tags":["x","y"],"active":true}"#;
    while buf.len() + record.len() + 2 < 1024 * 1024 {
        if buf.len() > 1 { buf.push(b','); }
        buf.extend_from_slice(record);
    }
    buf.push(b']');
    buf
}

fn parse_1mb(c: &mut Criterion) {
    let bytes = fixture_1mb();
    c.bench_function("parser/event_reader/1MB", |b| {
        b.iter(|| {
            let mut r = jfmt_core::EventReader::new(&bytes[..]);
            while let Some(ev) = r.next_event().unwrap() {
                black_box(ev);
            }
        });
    });
}

criterion_group!(benches, parse_1mb);
criterion_main!(benches);
```

Each `[[bench]]` entry in `crates/jfmt-core/Cargo.toml` declares
`harness = false` so criterion's own harness runs (not Rust's default
test harness).

## 5. Cargo.toml metadata template

Every `[package]` table gains:

```toml
description = "<crate-specific 1-line>"
keywords = ["json", "ndjson", "streaming", "cli"]    # max 5 for crates.io
categories = ["command-line-utilities", "parser-implementations"]
readme = "../../README.md"
homepage.workspace = true
documentation.workspace = true
```

Per-crate descriptions:

| Crate | description |
|---|---|
| `jfmt-core` | "Streaming JSON / NDJSON parser, writer, filter, validator at TB scale" |
| `jfmt-io` | "I/O adapters for jfmt: stdin/stdout, gzip/zstd compression" |
| `jfmt-cli` | "Streaming JSON / NDJSON formatter and filter CLI" |

`[workspace.package]` adds:

```toml
homepage = "https://github.com/jokerlix/XJsonView"
documentation = "https://docs.rs/jfmt-cli"
```

`jfmt-cli` may need a slightly different `keywords` set (e.g.
`["json", "jq", "ndjson", "cli", "formatter"]`) since CLI users search
differently than library users. Implementation chooses; both shapes
are valid for crates.io.

## 6. Errors and exit codes

M6 introduces no new runtime exit codes. Existing variants
(`Success` / `InputError` / `SyntaxError` / `SchemaError`) cover
everything jfmt does today.

cargo-dist's `release.yml` failures surface in the GitHub Actions UI
with non-zero step exit; not a jfmt CLI concern.

## 7. Testing strategy

### 7.1 Bench self-smoke

After each bench task (4, 5, 6) the implementer runs:

```bash
cargo bench --no-run    # must compile
cargo bench -- --quick  # criterion's quick mode: ~1s per bench (vs 5s default)
```

No throughput assertions; the goal is "harness compiles + runs without
panic." Real numbers are recorded to `target/criterion/` for human
inspection.

### 7.2 Metadata validation

After Task 3:

```bash
cargo metadata --no-deps --format-version=1 \
  | jq '.packages[] | {name, description, keywords, categories}'
```

Each crate must have non-null values for all four fields.

### 7.3 CHANGELOG sanity check

After Task 7 (D3 flattens 0.0.x history into a single table, so the
top-level `## [...]` headings are exactly two: Unreleased + 0.1.0):

```bash
grep -c -E "^## \[0\.1\.0\]|^## \[Unreleased\]" CHANGELOG.md
```

Must be exactly 2. The pre-1.0 history is verified separately by
checking the table contains all six tags:

```bash
grep -c -E "^\| v0\.0\.[1-6] " CHANGELOG.md
```

Must be 6.

### 7.4 cargo-dist plan dry-run

After Task 8:

```bash
cargo dist plan --output-format=human
```

Must list 5 targets and 2 installer scripts. Failure mode = config
mismatch; fix in `[workspace.metadata.dist]` and re-run.

### 7.5 release workflow YAML check

After Task 8 generates `release.yml`:

```bash
# If actionlint is on PATH; otherwise skip.
actionlint .github/workflows/release.yml || true
```

Optional. cargo-dist's generator is high-quality so this rarely
fires; it's a belt-and-braces step.

### 7.6 Manual verification (Task 11)

After `git tag v0.1.0 && git push origin v0.1.0`:

1. Watch GitHub Actions run `release.yml`. Expected: 5 platform
   matrix jobs run in parallel, all green; final job uploads to a
   new `v0.1.0` Release.
2. Visit `https://github.com/jokerlix/XJsonView/releases/tag/v0.1.0`.
3. Assert presence:
   - 5 tarballs (one per target)
   - 1 `jfmt-installer.sh`
   - 1 `jfmt-installer.ps1`
   - SHA256 / blake3 checksums file
4. Smoke-test one tarball locally:
   `curl -L <tarball-url> | tar -xz && ./jfmt --version`
   Expected: prints `jfmt 0.1.0`.

If anything fails, **fix-forward** (don't retag v0.1.0). Patch the
workflow / metadata, push a new commit, and repeat with `v0.1.1`.
Retagging is destructive and confuses downstream installers.

### 7.7 Existing test suite untouched

Every M6 task ends with `cargo test --workspace` + `cargo clippy
--workspace --all-targets -- -D warnings`. Both must stay green; M6
adds no new runtime behaviour, so any regression is a packaging mistake.

## 8. Risks and mitigations

1. **criterion MSRV 1.75.** Same drill as jaq + sysinfo + jsonschema.
   Plan Task 1 spikes the latest version that compiles on 1.75 and
   pins. criterion 0.5.x is the realistic target.

2. **cargo-dist generated workflow + GitHub Actions runner drift.**
   cargo-dist hardcodes runner labels (`ubuntu-22.04`, `macos-13`)
   that GitHub deprecates over time. Plan Task 8 spike captures the
   chosen cargo-dist version; if a runner gets deprecated later
   (months from now), Phase 1.5 regenerates with a newer cargo-dist.

3. **First release run failure.** Task 11 explicitly says fix-forward
   not retag. If `release.yml` fails, the tag stays; we patch and
   `v0.1.1`. No history rewriting.

4. **`aarch64-unknown-linux-gnu` build failures.** Cross-compile or
   GitHub's ARM runners may flake. If Task 11's verify shows aarch64
   missing, drop that target from `[workspace.metadata.dist].targets`,
   re-tag `v0.1.1`, file an issue to add it back later. Don't block
   Phase 1 close on aarch64.

5. **bench fixture allocation cost dominating bench time.**
   `fixture_1mb()` is called once per `bench_function`, so it's
   amortised across criterion's iteration loop. If criterion's stats
   show a big variance, move fixture into a `lazy_static` /
   `OnceLock` so it's truly constructed once. Defer until observed.

6. **`docs.rs` documentation builds.** `documentation =
   "https://docs.rs/jfmt-cli"` is aspirational — we don't publish to
   crates.io in M6, so docs.rs builds nothing yet. Link is correct
   for the Phase 1.5 publication; no harm in setting it now.

## 9. Acceptance criteria

- `cargo test --workspace` green.
- `cargo clippy --workspace --all-targets -- -D warnings` clean.
- `cargo bench --no-run` compiles all three bench files.
- `cargo bench -- --quick` runs without panic.
- `LICENSE-MIT` and `LICENSE-APACHE` exist at repo root with valid
  license text.
- `CHANGELOG.md` exists with `[0.1.0]` and pre-1.0 history.
- Every crate's `Cargo.toml` has `description`, `keywords`,
  `categories`, `readme` set.
- `cargo dist plan` reports 5 targets + 2 installers without error.
- README has CI badge, license badge, "Install (prebuilt binaries)"
  section.
- Phase 1 spec marks M6 shipped + Phase 1 closed.
- `git tag v0.1.0` exists and `git push --tags` succeeds.
- GitHub Release page for `v0.1.0` shows 5 tarballs, 2 installers,
  and a checksum file (manual verification).
